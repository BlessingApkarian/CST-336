.onCreate()

This callback must be implemented. It is activated when an app is opened. Once used, the app enters the Created. Inside .onCreate() your code is basic application startup logic that should occur only once for the entire life of the activity. For example, during implementation of onCreate(), you might:

  • bind data to lists
  • associate the activity with a ViewModel
  • instantiate some class-scope variables

    • .onDestroy()

    This callback is called before the activity is destroyed. The system invokes this callback either because:

  • the activity is finishing (due to the user completely dismissing the activity or due to finish() being called on the activity)
  • the system is temporarily destroying the activity due to a configuration change (such as device rotation or multi-window mode)

    • This is where the lifecycle components can clean up anything it needs to before the Activity is destroyed.

    .onStart()

    After the system is done entering Created it enters the Started state. The .onStart() method call makes the activity visible to the user, as the app prepares for the activity to enter the foreground and become interactive.

    .onStop()

    When your activity is no longer visible to the user, it has entered the Stopped state caused by the .onStop() method. This is when the app stops evrything to spare resources.

    .onPause()

    This callback is made when the user first leaves the activity. The app is no longer in the foreground. It may still be visible, however, if the user goes into the multi-window mode. Like in .onStop(), it pauses some functions and operations to save resources.

    .onResume()

    This method is invoked when the app comes out of the Paused State or Started Sate. Suring this lifecycle, any functionality the app contains is active.

    You can find examples of these callbacks at Android Developers
